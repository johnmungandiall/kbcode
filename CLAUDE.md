# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`kbcode` is a small, self-contained AI coding agent that runs in the terminal. It is itself built by blending ideas from five reference agents (cloned, for study only, into the gitignored `references/`): Claude Code (agentic loop + tools), Hermes (persistent memory/skills + context compaction), claude-kb (token-cheap `kb/` notes + `path:line` pointer checking), Kilo Code (modes), and openclaw (tool-call repair). When extending kbcode, the working assumption is that new capability comes from one of those concepts.

## Commands

- Install deps: `python -m pip install -r requirements.txt`
- Install as a command: `pip install -e .` (uses `pyproject.toml`; the `[project.scripts]` entry point `kbcode = "kbcode.cli:main"` puts a real `kbcode` on PATH, so `kbcode …` works anywhere — equivalent to `python -m kbcode …`). Version is `dynamic` from `kbcode.__version__`.
- End users install/update from GitHub: `pip install git+https://github.com/johnmungandiall/kbcode.git`; in-app `kbcode update` (`_self_update` in cli.py) shells out to `pip install --upgrade` on the same target.
- Show version: `kbcode --version` / `-v` / `-V` (handled before config/key load in `main`), the `/version` chat command, and the banner title. Single source: `kbcode.__version__`. A release = bump `__version__`, then tag `vX.Y.Z` + push.
- Run the chat REPL: `python -m kbcode`
- One-shot task: `python -m kbcode "do the thing"` (add `-y` / `--yes` to auto-approve writes and commands)
- Work on a different project: `python -m kbcode -C "<path>"` (`-C`/`--dir`/`--project` set the project root; the agent's tools are confined to it, so this is how you target another repo without `cd`). The folder must exist.
- Scaffold a project: `python -m kbcode init` (creates `AGENT.md`, the `kb/` note set, `.kbcode/`) — accepts a path: `python -m kbcode init "<path>"`, or `python -m kbcode -C "<path>" init`.
- Configure provider/model interactively: `python -m kbcode model` (picks provider, takes a key, **auto-fetches** the model list, saves **globally** to `~/.kbcode/settings.json` + `~/.kbcode/.env` so it applies to every project)
- There is **no test suite**. The de-facto check is byte-compilation: `python -m py_compile kbcode/*.py`. Logic is verified with throwaway inline scripts (construct `Tools`/`Agent`/`suggest()` and assert), not a framework.

### Windows / PowerShell notes (this is the dev environment)
- Set `PYTHONIOENCODING=utf-8` before running anything that prints the UI. The terminal uses emoji and em/box-drawing characters; the default cp1252 console raises `UnicodeEncodeError` on them. Output that flows through `rich` is usually fine; raw `print()` of those chars is not (which is why pointer-checker messages use ASCII `->`).
- Running `python -m kbcode` from another directory needs `PYTHONPATH` pointed at the repo root (the package isn't installed).

## Architecture — the big picture

The design goal is **provider-agnostic**: one agent loop drives Claude or any OpenAI-compatible model unchanged. Understanding the codebase means understanding how three things interact — the normalized message format, the provider translation layer, and the per-turn mode application.

### Normalized messages + the `raw` replay field (provider.py ↔ agent.py)
`Agent.messages` is a list of **normalized** items, never a provider's native shape:
- `{"role": "user", "content": str}` (plus an optional `"images": [{"media_type", "data"(base64)}]` for vision turns — see Image input below)
- `{"role": "assistant", "text", "tool_calls": [ToolCall], "raw": <native>}`
- `{"role": "tool_results", "results": [{"id", "content", "is_error"}]}`

Each provider (`AnthropicProvider`, `OpenAICompatibleProvider`) translates this to/from its own API in `_to_native`, and stores the model's own assistant payload back in `raw` so the next request replays it **losslessly** (Claude thinking blocks vs OpenAI `tool_calls` differ structurally). A session uses exactly one provider, so `raw` is always that provider's shape. **Invariant to preserve:** the normalized↔native translation must round-trip, and user/assistant turns must stay alternating after any message-list surgery (see compaction).

`get_provider(config, ui=None)` dispatches on `config.kind` (`"anthropic"` vs `"openai"`). Every non-Claude provider — OpenAI, Gemini, DeepSeek, OpenRouter, MiMo, custom — is the *same* `OpenAICompatibleProvider` with a different `base_url`. Anthropic calls use a staged-fallback (`thinking`/`output_config` → `thinking` → plain) that catches `TypeError` for older SDKs.

**Resilience (provider.py).** Every real API call is wrapped in `_with_retry(fn, ui)`: transient failures (HTTP 429, 5xx, connection/timeout — classified by `_classify` from the SDK-agnostic `status_code` + message, so it needs neither SDK's exception types) are retried with exponential backoff (`_MAX_RETRIES`/`_BACKOFF_BASE`), emitting a yellow "retrying…" notice through the optional `ui`. Hard errors (401/403 auth, 4xx bad request) raise `ProviderError(message, hint=…)` immediately — no retry. `_with_retry` deliberately **re-raises `TypeError` untouched** so the Anthropic staged-fallback still works. The CLI catches `ProviderError` (REPL turn loop *and* one-shot `main`) to print a clean message + hint instead of a traceback, and catches `KeyboardInterrupt` to drop back to the prompt (exit 130 in one-shot).

**Interrupt a turn (interrupt.py).** `agent.run` is wrapped in `interrupt_on_escape()` (REPL + one-shot): a daemon thread watches the keyboard and, on **Esc**, calls `_thread.interrupt_main()` — surfacing as the same `KeyboardInterrupt` the CLI already handles (Ctrl-C still works). TTY-only and fully guarded (no-op when piped / unsupported); Windows uses `msvcrt`, POSIX uses `termios`+`select` (distinguishing a lone Esc from arrow-key escape sequences). For Esc to land **mid-request** (not just between steps), `Agent._complete` runs the blocking `provider.complete` on a daemon worker and waits on it in short Python-level polls (`done.wait(0.05)`) — otherwise a `KeyboardInterrupt` is swallowed by the in-flight socket read until it returns. With the poll, the interrupt is delivered within ~50 ms; the orphaned worker just finishes and its result is dropped (provider errors raised in the worker are carried over and re-raised on the main thread, so `ProviderError` handling is unchanged). The spinner shows an "(Esc to interrupt)" hint.

**Image input / vision (images.py + prompt_input.py + provider.py).** A user turn may carry attachments: `messages[i]["images"]` is a list of `{"media_type", "data"(base64)}` dicts (`images.py` builds them — `grab_clipboard_image()` via Pillow, or `load_image_file(path)`, normalizing/downscaling to ≤1568px PNG when needed; both guarded so a missing Pillow / empty clipboard returns `None`). The chat input (`make_input`) binds **Alt+V** to grab the clipboard image into a pending buffer shown in the bottom toolbar; `pop_images()` drains it and `cli._repl` passes the list to `agent.run(user, images=…)` (also `/image [path]` in chat and `--image/-i <path>` one-shot). `Agent.run` only adds the `images` key when non-empty, so non-vision turns are byte-identical to before. Each provider's `_to_native` expands an image-bearing user message into its own vision format — Anthropic `{"type":"image","source":{base64}}` blocks, OpenAI `{"type":"image_url","image_url":{"url":"data:…"}}` parts — and a text-free image-only turn is valid. `compaction.estimate_tokens` counts each image as a flat ~1300 tokens instead of its base64 length, so a screenshot doesn't trigger needless compaction. Needs a vision-capable model; Pillow is a dependency (clipboard path only).

### Modes apply per turn (modes.py + agent.py)
A `Mode` pairs instruction text with an allowed-tool set. `Agent.run` rebuilds two things on **every** model call from the current mode: `_system_for_mode()` (base system prompt + mode instructions) and `_mode_schemas()` (tool schemas filtered to what the mode allows). Enforcement is two-layer: disallowed tools are never shown to the model, *and* `run()` guards at execute time. Built-ins: `code`/`debug` (all tools), `architect` (read + notes), `ask` (read-only). Custom modes load from `.kbcode/modes/*.md`. Note `manage_todos` lives in the `READ` group so the task checklist is available even in read-only modes.

### Subagents delegate into a fresh context (subagents.py + agent.py + tools.py)
`load_subagents` reads `.kbcode/agents/*.md` (same frontmatter parser as modes — `description:`, `tools:`) into `Subagent` records. The wiring is a small back-reference: `Agent.__init__` sets `tools.subagents` and `tools.delegate = self._run_subagent`. `Tools.schemas` then **conditionally** appends the `run_subagent` schema (only when subagents exist), with the roster baked into its description. When the model calls `run_subagent`, `_tool_run_subagent` calls back through `tools.delegate`; `Agent._run_subagent` runs a separate bounded loop (`_SUBAGENT_MAX_STEPS`) with the subagent's own system prompt + filtered schemas, shares the same `Tools` (so file/KB side-effects land in the same project), blocks nested delegation, and returns only the final text. Subagent token usage still accrues to `Agent.usage`.

### Standing orders + usage tally
`build_system_prompt` takes an optional `standing_orders` string (from `.kbcode/standing-orders.md`) injected right after the base rules; `cli._build_agent` ignores the untouched scaffold template so its examples never become live orders. `Agent.usage` accumulates `requests`/`input_tokens`/`output_tokens` from each `LLMResponse.usage` (both providers populate it); `Agent.insights()` + `pricing.estimate_cost` back `/insights`. After every user turn `Agent._turn_summary` prints the `actions · tokens · elapsed` footer.

### Config precedence (config.py)
`load_config` resolves provider/model/base_url (and the API key, via `.env`) as **env vars > the project's `.kbcode`/.env > the launch folder's > the global `~/.kbcode` (`global_dir()`) > preset defaults**. The launch-folder and global fallbacks are what let you configure kbcode once and then `-C` it at any project without re-entering the key there; `python -m kbcode model` writes its choices to `~/.kbcode`. `PRESETS` is the source of truth for built-in providers. `Config` derives all paths (`kbcode_dir`, `kb_dir`, `memory_db`, `agent_md`, `settings_file`, `standing_orders_file`) from `project_dir`. Because every path is a property off `project_dir`, the project can be retargeted: the CLI picks it via `-C/--dir/--project` or `init <path>` (parsed by `_take_dir` in cli.py), and the in-chat `/open <folder>` mutates `config.project_dir` then rebuilds `kb`, `memory`, and the agent (re-scaffolding the new folder, keeping the same provider/model/key). A REPL guard catches the common slip of typing the terminal-only `init`/`model` as a chat message and points at `/open` instead.

### Context stays cheap two ways
- **Compaction (compaction.py):** when `estimate_tokens(messages)` crosses `compact_threshold`, the middle exchanges are summarized into one recap spliced onto the first kept tail turn, protecting the first + last exchanges and preserving alternation. Auto in `Agent.run`; manual via `/compact`.
- **Knowledge base (knowledge_base.py):** `kb/` holds short notes loaded into the system prompt so the agent doesn't re-scan files. `check_pointers()` (`/kb-check`) resolves every `path:line` reference and flags missing files / stale line numbers; placeholder examples are skipped. `fix_pointers()` (`/kb-check --fix`) goes further: it relocates a drifted pointer by the **code symbol** named on the same note line (`_anchors` + `_relocate` — prefer a unique definition line, then a unique call, then a unique mention), and reports what it couldn't safely move.

### Tools, repair, and the UI seam
- **Tool-call repair is two layers (openclaw idea).** *Parse layer* (`repair.py`): if the model returns text but **no** structured `tool_calls`, `promote(text, allowed_names)` scans that text for a tool call the model wrote as plain text — `[read_file]\n{…}`, `[tool:name]`, `<name>{…}</name>`, or a bare `{"name"/"tool"/"function", "arguments"/…}` object — and recovers `(name, args)` pairs (only for names the mode actually offers, so prose/JSON is never mis-promoted). `Agent.run` (and the subagent loop) then runs them via `_run_promoted` and feeds the outputs back as a **plain `user` turn** (not `tool_results`, since there are no native tool ids to replay) with a nudge to use the real format — so a weaker model that skips the tool interface keeps going instead of stalling at "no tool calls → done". *Execute layer*: `Tools.execute` runs a `_repair()` step first — unknown tool name → closest match via `difflib` + the tool list; missing required args → names them. Together they let weaker models self-correct instead of hard-failing. Path access is confined to the project root via `_resolve()`, and `_protected_reason()` is a second safety rail (Hermes `file_safety` idea) that **refuses** `write_file`/`edit_file` to `.git/`, `.ssh/`, `.env` & secrets, private keys (`*.pem`/`*.key`/…), and kbcode's own state (`.kbcode/memory.db`, `settings.json`) — while allowing templates (`.env.example`), `.gitignore`, and user-authored `.kbcode/agents`/`modes` markdown. `write_file`/`edit_file`/`run_command` then gate through `Permissions`, and the write/edit tool-lines show the **full resolved path** (so the user sees where a file lands). The tool roster is `read/write/edit/list/search/run` + `kb_read/kb_write` + `remember/recall/save_skill` + `manage_todos` (in-memory checklist, rendered by `/todo`) + the conditional `run_subagent`.
- All terminal output goes through `TerminalUI` (ui.py) — the loop never calls `console.print` directly. Tool calls render as a human verb+target via `_describe_tool`; results indent under them. `prompt_input.py` provides both the `/` autocomplete popup and `select()` (an inline arrow-key menu). Both are TTY-only and fully guarded: `make_input` returns `None`, and `select()` returns `(False, None)`, when piped / no console / lib missing — so they degrade to plain readers and tests/pipes never break.
- **Permissions** (`permissions.py`) hold an `always_allow` set and call `ui.permission(tool, detail)`, which renders a yellow context panel then offers a **selectable Yes / Always / No menu** through `select()`, falling back to a typed `y/N/a` prompt (`_permission_typed`) when no menu is available. `Permissions(ui=None)` keeps an ASCII-only `_plain` path for headless use.

## When adding things

- **A new tool:** add its schema to `Tools._base_schemas` *and* a `_tool_<name>` method; if it should be restricted, add it to the right group (`READ`/`NOTES`/`EDIT`/`EXEC`) in modes.py, otherwise it's implicitly available everywhere. (A schema that depends on runtime state — like `run_subagent` — is appended in the `schemas` property instead.)
- **A new slash command:** add it to `COMMANDS` in ui.py (single source for both `/help` and the autocomplete popup; grouping for `/help` lives in `TerminalUI.help`) and handle it in `_repl` (cli.py). Argument completion comes from the `arg_options` map passed to `make_input`. Pass any command label with `[...]`/`<...>` hints to the UI as a `Text` object, not raw markup.
- **A new subagent or mode:** ship a markdown file (`.kbcode/agents/*.md` or `.kbcode/modes/*.md`) with `description:`/`tools:` frontmatter — no code change needed. A starter `code-explorer` subagent is scaffolded by `_scaffold`.
- **An interactive picker:** reuse `prompt_input.select()` (returns `(available, index)`); always handle `available is False` with a non-interactive fallback, as `TerminalUI.permission` does.
- **A new provider:** prefer adding a `PRESETS` entry; only write a new `LLMProvider` subclass if it isn't OpenAI-compatible.

## Boundaries

- `references/` is cloned third-party source for studying concepts — **gitignored, not part of the product**. Never import from it or ship its code.
- `.kbcode/` (memory db, settings) and `.env` are gitignored, per-machine/secret. The README and `.env.example` document every provider and tuning env var.
